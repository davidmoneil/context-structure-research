# Deep Research: Obsidian Desktop Release Notes & Changelog (Oct 2025 - Feb 2026)

**Research Date**: 2026-02-16
**Confidence Level**: High
**Sources Consulted**: 14

## Executive Summary

Obsidian Desktop is currently at **v1.12.1** (released February 10, 2026). Since October 2025, the application has undergone four major version bumps (1.9 -> 1.10 -> 1.11 -> 1.12), each introducing significant platform capabilities. The most impactful changes for integration/automation work are:

1. **Official CLI** (v1.12) -- 100+ commands for terminal-based vault management, including properties, files, search, templates, and a `dev:eval` JavaScript execution command.
2. **Markdown links in properties** (v1.11) -- Properties now support internal markdown links with automatic update on file rename/move.
3. **Canvas backlinks in Graph view** (v1.12) -- Canvas files are now first-class citizens in the linking graph.
4. **Bases maturation** (v1.9-1.12) -- Database views over vault files with YAML `.base` format, filter/formula engine, and plugin API for custom views.
5. **SecretStorage API** (v1.11) -- Shared credential management across plugins.

These changes substantially improve the viability of programmatic Obsidian integration for canvas management, frontmatter manipulation, templating, and query-based workflows.

### 2. Official CLI (v1.12) -- Major New Capability

The CLI is bundled with the Obsidian desktop app (not a separate install). It is the same binary as the GUI application.

**Binary Location**:
- macOS: `/Applications/Obsidian.app/Contents/MacOS/Obsidian`
- Linux/Windows: Same Obsidian executable, must be added to PATH

**Command Structure**: `obsidian [options] [command]`

**Primary Command Categories** (100+ commands):

| Category | Examples | Description |
|----------|----------|-------------|
| `files` | `files list`, `files read`, `files write` | Note content and file management |
| `search` | `search content`, `search path` | Full-text and path search |
| `properties` | `properties read`, `properties set` | Frontmatter/metadata CRUD |
| `tasks` | `tasks all`, `tasks pending` | Checkbox/task operations |
| `templates` | `templates list`, `templates apply` | Template management |
| `plugins` | `plugins list`, `plugins versions` | Plugin administration |
| `tags` | `tags all` | Tag operations |
| `bookmarks` | | Bookmark operations |
| `daily` | | Daily note operations |
| `links` | | Link operations |
| `sync` | | Sync operations |
| `themes` | | Theme operations |
| `vault` | | Vault info (name, path, counts) |
| `dev:eval` | | **JavaScript execution** |

**Global Options**:
- `-v, --version` -- Version info
- `-h, --help` -- Help
- Vault parameter must be the first argument if included

**TUI Mode**: Running `obsidian` without arguments launches an interactive terminal UI with navigation, file CRUD, and search.

**Availability Note**: Currently requires Catalyst License ($25 one-time) for Early Access. Will be free for all users later.

**Key Integration Implications**:
- `properties read`/`properties set` enables external frontmatter management
- `files read`/`files write` enables programmatic note creation
- `search content` enables vault-wide search from scripts
- `templates apply` enables template-based note creation from CLI
- `dev:eval` enables arbitrary JavaScript execution against the vault
- `daily` enables daily note automation

### 3. Properties/Frontmatter Changes

**v1.11.4 -- Markdown Links in Properties** (January 12, 2026):
- Text and list property types now support internal markdown links: `[Display](path/to/note.md)`
- Internal links are **automatically updated** when the destination file is moved or renamed
- This was a long-requested feature (community plugin `frontmatter-markdown-links` previously filled the gap)
- Previously only wikilinks `[[note]]` worked in properties

**v1.11.4 -- Daily Note Format Selection**:
- Daily note format can now be selected from predefined list
- Folder path normalization for daily notes

**v1.11.5 -- Default File to Open**:
- New setting: "Default file to open" with options: Last opened / New note / Specific / Daily

**v1.12 -- CLI Properties Commands**:
- `obsidian properties read` and `obsidian properties set` enable external frontmatter management
- Combined with `files read`/`files write`, full programmatic property manipulation is possible

### 4. Canvas Format Changes

**v1.12.0 -- Canvas Backlinks in Graph View**:
- Backlinks in Canvas files are now detected and shown in the backlinks view
- Canvas links are counted as links in the Graph view
- This makes canvas files first-class participants in the vault's link graph
- Previously, canvas files were isolated from the linking system

**JSON Canvas Spec (unchanged at v1.0)**:
The `.canvas` file format remains JSON Canvas 1.0. No spec changes since March 2024.

```json
{
  "nodes": [
    {
      "id": "required-string",
      "type": "text|file|link|group",
      "x": 0, "y": 0,
      "width": 400, "height": 300,
      "color": "optional - hex or preset 1-6",
      // Type-specific fields:
      "text": "markdown content",        // text nodes
      "file": "path/to/note.md",         // file nodes
      "subpath": "#heading",             // file nodes (optional)
      "url": "https://...",              // link nodes
      "label": "Group Name",            // group nodes (optional)
      "background": "path/to/img",       // group nodes (optional)
      "backgroundStyle": "cover|ratio|repeat"  // group nodes (optional)
    }
  ],
  "edges": [
    {
      "id": "required-string",
      "fromNode": "node-id",
      "toNode": "node-id",
      "fromSide": "top|right|bottom|left",  // optional
      "toSide": "top|right|bottom|left",    // optional
      "fromEnd": "none|arrow",              // optional, default: none
      "toEnd": "none|arrow",               // optional, default: arrow
      "color": "optional",
      "label": "optional edge label"
    }
  ]
}
```

**Color Presets**: `"1"`=red, `"2"`=orange, `"3"`=yellow, `"4"`=green, `"5"`=cyan, `"6"`=purple (or hex string)

**Canvas Properties Status**: Still no native frontmatter/properties support for `.canvas` files themselves. This remains a community feature request. The Advanced Canvas plugin offers this via a `.canvas.meta.md` companion file approach.

### 5. Bases -- Database Views Over Vault Files

**Timeline**:
- v1.9 (Aug 2025): Initial Bases release
- v1.10 (Oct 2025): Map view, List view, Group by, Summaries (totals/averages/counts), Bases API, CSV import
- v1.12 (Feb 2026): Search toolbar for filtering within Bases

**File Format**: `.base` files are **YAML** (not JSON). Can also be embedded in markdown code blocks.

**Core YAML Schema**:
```yaml
filters:
  # Global filters (apply to all views)
  - file.ext == "md"
  - file.inFolder("Projects")

formulas:
  # Computed properties
  price_per_unit: price / quantity
  days_until_due: deadline - now()

properties:
  # Display config for properties
  status:
    displayName: "Task Status"

views:
  - type: table       # table | map | list
    name: "My View"
    filters:           # View-specific filters
      - status != "done"
    # Type-specific fields (e.g., lat/long for map)
```

**Filter Syntax** (full expression language):

| Category | Examples |
|----------|----------|
| Comparison | `status == "done"`, `priority != 0`, `score >= 80` |
| Boolean | `!completed`, `urgent && !done`, `a \|\| b` |
| Arithmetic | `price + tax`, `total / count`, `value % 2 == 0` |
| Date | `file.mtime > now() - "7d"`, `deadline - now()` |
| String | `name.contains("project")`, `title.startsWith("Draft")` |
| List | `tags.contains("work")`, `links.isEmpty()` |
| File | `file.hasTag("project")`, `file.inFolder("Notes")`, `file.hasLink(this)` |

**Implicit File Properties**:
- `file.name`, `file.path`, `file.folder`, `file.ext` (string)
- `file.size` (number, bytes)
- `file.mtime`, `file.ctime` (date)
- `file.tags`, `file.links`, `file.embeds` (list)

**Special `this` Reference**: Contextual -- refers to base file, containing note, or active file depending on where the base is rendered.

**Bases API for Plugins** (v1.10+):
```javascript
registerView('my-view-id', {
  name: 'View Name',
  icon: 'my-icon',
  factory: () => new MyView(),
  options: (config: BasesViewConfig) => ({
    // view options
  })
});
```

**Roadmap items** (planned, not shipped):
- Calendar view for Bases
- Kanban view for Bases
- Bases support for Publish

### 6. Plugin API Changes

**v1.10 -- Bases API**:
- New `BasesViewRegistration` interface for plugins to register custom Bases view types
- Example: obsidian-maps plugin adds geographic map view

**v1.11.4 -- SecretStorage API**:
- `SecretStorage` and `SecretComponent` exposed to plugin API
- Shared credential store across plugins (e.g., API keys)
- Plugins can fetch keys from shared collection instead of per-plugin configuration
- New "Keychain" settings section in UI

**v1.11.4 -- SettingGroup API**:
- New `SettingGroup` class for organizing plugin settings with headers and logical grouping

**v1.11.4 -- Electron v39.2.6**:
- Enables Wayland support on Linux
- Inter Variable font support added

**v1.12.0 -- Breaking Changes**:
- `BaseOption#shouldHide` no longer receives config argument
- Must use `BasesViewRegistration.options` instead
- CSS variable rename: `--blur-translucency-s/m` -> `--blur-opacity-s/m`

**v1.12.0 -- CSS Changes**:
- New `corner-shape` CSS property (requires Chromium 139+)
- macOS: `--traffic-lights-offset-x/y` CSS variables for window button positioning

**v1.12.0 -- URI Enhancements** (v1.11.4):
- New `paneType` parameter for `new`, `open`, `daily` URI actions
- Values: `tab`, `split`, `window`
- New `unique` URI action (v1.12)

### 7. Markdown Extensions

**No new markdown syntax extensions** have been added in this period. The changes are behavioral:

- v1.12: Image resizing in Live Preview (drag corners, double-click to reset)
- v1.12: Image drag size no longer treated as display text
- v1.12: Blockquote spacing and bold link styling fixes
- v1.12: Reference link validation (no spaces before definitions)
- v1.11: URL paste converts selected text to Markdown link automatically
- v1.12: `daily:prepend` command now inserts after frontmatter (not at file start)

### 8. File Format Changes

**No breaking file format changes**:
- `.md` files: No format changes
- `.canvas` files: Still JSON Canvas 1.0 spec
- `.base` files: YAML format (stable since v1.9 introduction)
- `.json` config files: No changes

## Best Practices for Integration

Based on these findings, recommended approaches for an Obsidian integration/skill:

1. **Use the CLI for automation** -- The official CLI (`obsidian properties read/set`, `files read/write`, `search content`, `templates apply`) is now the preferred path for external tooling. It replaces the need for direct file manipulation in many cases.

2. **Properties/frontmatter management** -- Continue using YAML frontmatter in `.md` files. Markdown links in properties are now supported natively. Use `[[wikilinks]]` or `[text](path)` format.

3. **Canvas file manipulation** -- Direct JSON manipulation of `.canvas` files remains the approach. The format is stable at JSON Canvas 1.0. New: canvas links now appear in the graph, so canvas-created links have more visibility.

4. **Bases queries** -- `.base` files are YAML with a well-defined schema. They can be created programmatically. The filter/formula syntax is documented and stable.

5. **Template application** -- The CLI `templates apply` command automates what was previously GUI-only.

6. **Watch for Bases API expansion** -- Calendar and Kanban views are planned. The `BasesViewRegistration` API is the extension point.

## Common Pitfalls

- CLI requires Catalyst license (Early Access) as of Feb 2026
- CLI vault parameter must be FIRST argument (v1.12.1 fix note)
- `BaseOption#shouldHide` breaking change if writing Bases view plugins
- Canvas files still lack native properties/frontmatter (use companion file pattern)
- `daily:prepend` inserts after frontmatter, not at file start (changed in v1.12.1)

## Recommendations

1. **Immediately**: Test the Obsidian CLI on your system. Add the binary to PATH. Verify `obsidian properties read` and `obsidian files list` work against your vault.

2. **Short-term**: Update any canvas manipulation code to account for canvas backlinks now appearing in the graph. This means canvas file nodes that reference real files create real bidirectional links.

3. **Medium-term**: Consider migrating Dataview queries to Bases `.base` files. The YAML format is more structured and programmatically manageable than DQL code blocks.

4. **Long-term**: Watch the Bases API for Calendar and Kanban view types. These will enable richer programmatic views.
